验证本轮“下载解析提速”改造在本机 Docker 环境的实际效果，重点关注：

- `POST /api/download/parse` 的时延（P50 / P90）
- 抖音与 B 站的阶段耗时分解

### 4.1.2 抖音冷启动并发（uncached）

在重启 backend 后，对 4 条抖音样例同时发起解析：

- 单请求耗时：`2297ms / 4430ms / 4623ms / 7472ms`
- 总 wall time：`7473ms`
- 后端日志显示队列等待分别约：`0ms / 1ms / 1671ms / 3720ms`

结论：

1. 当前版本在并发下仍存在排队（由有界并发 + 最小间隔触发），但高峰总耗时已显著短于“旧版 10s 节流 + 全串行”的理论耗时。
2. 按旧参数粗估，4 个并发请求总耗时通常会落在 `30s+`，本次实测 `7.5s`，总时长降幅约 `75%` 量级（估算）。

## 5. 与改造前目标基线对照（估算）

> 说明：以下“改造前”来自实施方案中的历史经验区间，不是同批样本的 A/B 回放。

### 抖音

- 改造前目标基线：P50 `4.5~6.5s`，P90 `9~15s`
- 当前实测：P50 `2.477s`，P90 `2.524s`
- 估算降幅：
  - P50：约 `45% ~ 62%`
  - P90：约 `72% ~ 83%`

### B 站

- 改造前目标基线：P50 `3~5s`，P90 `6~10s`
- 当前实测：P50 `0.420s`，P90 `0.566s`
- 估算降幅：
  - P50：约 `86% ~ 92%`
  - P90：约 `91% ~ 94%`

## 6. 结论与后续

1. 当前版本在本机 Docker 环境下，解析时延已明显下降，且成功率维持 100%（本次样本）。
2. 抖音剩余主要耗时集中在 `ratioProbe`（约 1s），后续可重点评估：
   - 跨质量目标并发探测（在风控约束内）
   - 进一步把“线路优化”后移到 `get-url`，parse 只负责补齐画质集合
3. 建议再补一轮高并发压测（如 20/50 并发）验证排队与风控边界下的 P95/P99。
