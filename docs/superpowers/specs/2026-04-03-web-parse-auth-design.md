# Web 端解析鉴权收紧设计

## 目标

- 收紧产品权限：未登录用户在 Web 端不能发起视频解析。
- 收紧接口权限：`POST /api/download/parse` 改为必须携带 JWT，不能再被匿名调用。
- 保持已登录用户现有解析、预览、下载链路不变。
- 避免只改前端导致的接口绕过风险。

## 当前行为

- Web 首页 `Home` 中，未登录用户仍可点击“解析”，前端会直接请求 `/download/parse`。
- Web 下载按钮已有限制：未登录时会提示“请先登录后再下载”并跳转登录页。
- 后端 `DownloadController.parseVideo` 当前是公开接口，没有 `JwtAuthGuard`。
- Mobile 端和静默下载也复用 `/download/parse`，但它们的 API 客户端会自动附带 token，天然符合鉴权要求。

## 约束

- 不改变“下载仍需登录”的既有规则，只补齐“解析也需登录”。
- 不为 Web 和 Mobile 拆分两套解析接口，避免权限模型分叉。
- 不修改解析成功后的数据结构，尽量把影响面限制在鉴权和错误反馈上。
- 本轮优先做权限收口和回归测试，不扩展新的会员分级或配额策略。

## 方案对比

### 方案 A：前后端一起收紧鉴权

- 前端未登录时直接拦截解析操作，提示并跳转登录页。
- 后端给 `/download/parse` 增加 `JwtAuthGuard`，匿名请求直接返回 `401`。
- 优点：UI 和接口都被封住，权限模型一致，最符合目标。
- 缺点：需要同步更新 Web 用例，并确认移动端已登录流不受影响。

### 方案 B：只收紧后端接口

- 后端强制鉴权，前端不做预拦截，未登录用户点击解析后由接口返回 `401`。
- 优点：安全性足够，服务端是最终真相。
- 缺点：Web 用户体验较差，会先触发请求再失败，页面反馈也不够明确。

### 方案 C：只收紧 Web 前端

- Web 未登录时不让点解析，但接口仍然公开。
- 优点：改动最小。
- 缺点：不满足“解析接口也要加鉴权”，可以被直接绕过。

## 采用方案

采用方案 A。

- Web 端在发起解析前先检查登录态，未登录则设置明确错误文案并跳转 `/login`。
- 后端 `DownloadController.parseVideo` 增加 `JwtAuthGuard`，与 `get-url`、`create-task` 保持一致。
- Mobile 端继续沿用当前认证模型，不做特殊分支；未登录时若触发解析，会收到统一的 `401` 语义。

## 设计细节

### Web 端

- 在首页解析入口增加登录态前置判断，未登录时不调用 `useVideoParser().parse`。
- 错误文案与下载限制保持同一口径，使用“请先登录后再解析”。
- 触发拦截时跳转到 `/login`，减少用户在首页停留后重复尝试。
- 已登录用户的解析交互、预览数据、画质选择和后续下载逻辑保持不变。

### 后端接口

- 给 `POST /download/parse` 增加 `@UseGuards(JwtAuthGuard)`。
- 保持现有控制器内部解析逻辑、运行时埋点和错误归一化逻辑不变。
- 未携带或携带无效 JWT 的请求交由现有鉴权体系返回 `401`。

### 共用客户端影响

- Web：新增本地预拦截，避免未登录时发起无意义请求。
- Mobile：首页解析和静默下载会继续通过现有 axios request interceptor 自动带 token；已登录行为不受影响。
- 其他复用 `/download/parse` 的已登录页面，例如下载历史重新下载，维持现状。

## 测试策略

- 前端补首页行为测试：
  - 未登录点击解析时，不发起 `/download/parse` 请求。
  - 未登录点击解析时，展示“请先登录后再解析”并跳转 `/login`。
- 后端补控制器鉴权测试：
  - `parseVideo` 路由携带 `JwtAuthGuard` 元数据。
  - 既有解析成功和解析失败行为测试继续保留，避免功能回退。

## 验证标准

- Web 未登录用户无法通过页面解析视频。
- 匿名请求直接访问 `POST /api/download/parse` 会被后端拒绝。
- 已登录 Web 用户的解析、预览、下载流程不回退。
- 已登录 Mobile 用户的解析与静默下载流程不因为本次鉴权收紧而失效。
